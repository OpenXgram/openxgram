Vault

Vault — 4단계 보안

Vault — four lines of defense

1 · Disk

AES-256-GCM 암호화, scrypt KDF (N=2^17). seed.enc + vault.db 분리 저장.

AES-256-GCM at rest, scrypt KDF (N=2^17). seed.enc and vault.db kept separate.

2 · Field

각 secret에 고유 nonce + ACL 태그. 같은 DB 접근 권한도 필드 단위로 차단.

Per-secret nonce + ACL tag. Even with DB access, fields stay locked.

3 · Transport

peer 간 통신은 Noise/TLS. 모든 메시지에 secp256k1 서명 → 위변조 차단.

Peer comms over Noise/TLS, every message secp256k1-signed.

4 · In-memory

secret 객체는 zeroize on drop, mlock으로 swap 차단, no-debug derive.

Secrets zeroize on drop, mlock prevents swap, no Debug derives.

Audit

감사 체인

Audit chain

모든 vault 작업과 결제는 append-only 로그에 기록되며, 각 엔트리는 이전 엔트리의 해시를 포함합니다 (단방향 체인).

Every vault op and payment is appended to a log; each entry includes the previous entry's hash (hash chain).

Stack

Tauri 데스크톱 스택

Tauri desktop stack

  • 코어 데몬 — Rust 단일 바이너리, IPC over local socket
  • TUI — ratatui, 헤드리스 머신/SSH 환경에서 즉시 사용
  • GUI (Phase 2) — Tauri + React, 같은 IPC 클라이언트 재사용
  • 저장소 — SQLite + sqlite-vec, OS 키체인은 seed.enc 보호용으로만
  • 전송 — IPC → Tailscale → XMTP → Nostr 단계적 확장
  • Core daemon — single Rust binary, IPC over local socket
  • TUI — ratatui, ready for headless / SSH
  • GUI (Phase 2) — Tauri + React reusing the same IPC client
  • Storage — SQLite + sqlite-vec; OS keychain only protects seed.enc
  • Transport — IPC → Tailscale → XMTP → Nostr, layered
