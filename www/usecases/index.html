Use cases

현실 시나리오 다섯 가지.

Five real-world scenarios.

각 사례는 시나리오 → 명령 시퀀스 → 기대 결과 → 주의점으로 구성됩니다.

Each case: scenario → commands → expected result → caveats.

Case 1

여러 머신에서 동일 ChatGPT 세션 이어가기

Continue the same ChatGPT session across machines

시나리오

Scenario

회사 노트북에서 시작한 대화를 집 데스크톱에서 그대로 이어가고 싶다. 모델·UI는 달라도 메모리는 같아야 한다.

Start a conversation on your work laptop, pick it up at home — different model or UI, same memory.

명령

Commands

# laptop
$ oxg say "draft Q3 report intro"
$ oxg sync push --to home-desktop

# desktop
$ oxg sync pull --from laptop
$ oxg recall --episode latest
[L1 ep-042] 12 messages — last: "draft Q3 report intro"
$ oxg say "continue from where we stopped"

기대 결과

Expected result

  • 두 머신의 episode 타임라인이 머지됨 (서명 충돌 없음)
  • 벡터 검색이 양쪽 메시지를 모두 찾음
  • Episode timelines merged (no signature conflicts)
  • Vector search finds messages from both sides

주의점

Caveats

  • peer를 먼저 trusted로 등록해야 sync가 자동 진행됩니다.
  • The peer must be registered as trusted before sync auto-runs.
Case 2

Vault로 API 키 안전 공유 (마스터 승인 정책)

Share an API key safely via Vault (master approval)

시나리오

Scenario

Eno(코딩 에이전트)가 OpenAI API를 써야 한다. 키를 평문으로 넘기지 않고, 호출마다 정책 검증을 거치게 한다.

Eno (the coding agent) needs OpenAI API access. Never hand it a plaintext key — gate each call by policy.

명령

Commands

$ oxg vault put openai.api_key --acl "agent:eno" --tier confirm
$ oxg vault policy set openai.api_key \
    --daily-limit 200 --machine kai-mbp --machine kai-desktop
$ oxg vault audit openai.api_key --since 1d
[2026-05-04 09:11 KST] eno  GET ok   (sig 0x9a1c..)
[2026-05-04 09:42 KST] eno  GET deny (over daily-limit)

기대 결과

Expected result

  • Eno가 키를 직접 보유하지 않고, 매 호출 vault에 요청 → 정책 통과 시만 노출
  • 한도 초과 시 confirm 프롬프트가 마스터에게 도달
  • Eno never holds the key; it asks vault each call, only released if policy passes
  • Past the cap, a confirm prompt is routed to the operator

주의점

Caveats

  • tier=auto는 편하지만 감사만 남고 차단은 없으니, 외부 API에는 confirm 이상을 권장.
  • tier=auto only logs — never use it for outbound APIs; prefer confirm or higher.
Case 3

USDC 결제 자동화 (한도 + MFA)

USDC payments with limits + MFA

시나리오

Scenario

리서치 에이전트가 매일 자료 구독료를 결제한다. 잔돈 수준은 자동, 큰 금액은 MFA로.

The research agent renews subscriptions daily. Pennies auto-pay, larger amounts require MFA.

명령

Commands

$ oxg pay policy set --currency usdc --network base \
    --auto-under 5usdc --confirm-under 50usdc --mfa-over 50usdc \
    --daily-cap 200usdc

$ oxg pay send 0xPAPER... 3.20usdc --memo "arxiv pro renew"
[auto] tx 0x9c3a... confirmed in 2 blocks
[memory] L2 receipt saved (kind=fact)

$ oxg pay send 0xVENDOR... 80usdc
[mfa] open authenticator → enter code: ___

기대 결과

Expected result

  • 모든 영수증이 L2 memory의 fact로 자동 저장 → "지난 달 구독비"를 자연어로 검색 가능
  • Receipts persist as L2 facts; ask "last month subscription spend" in natural language

주의점

Caveats

  • 최초 1회 카운터파티는 confirm으로 풀고 화이트리스트에 등록할 것.
  • First-time counterparties: confirm once, then add to allowlist.
Case 4

Nostr cross-network 메시지

Cross-network messaging via Nostr

시나리오

Scenario

다른 클라이언트(Damus 등)를 쓰는 친구에게 OpenXgram에서 직접 메시지를 보낸다. 같은 키, 다른 표면.

Send a DM from OpenXgram to a friend on Damus. Same key, different surface.

명령

Commands

$ oxg peer add nostr://npub1friend...
$ oxg relay add wss://relay.damus.io
$ oxg send --to nostr://npub1friend... "hey, ratchet test"
[ok] event 0xa12... published to 3 relays
[ratchet] sk rotated (epoch 7 → 8)

기대 결과

Expected result

  • 친구의 일반 Nostr 클라이언트에서도 수신됨 (NIP-44 호환)
  • OpenXgram 측에는 L0 message + 서명 + ratchet epoch 기록
  • Friend's stock Nostr client receives it (NIP-44 compatible)
  • OpenXgram side records L0 message + signature + ratchet epoch

주의점

Caveats

  • 릴레이를 자체 호스팅하면 검열 저항이 더 강해집니다 (PRD-NOSTR-06).
  • Self-hosting a relay raises censorship resistance (PRD-NOSTR-06).
Case 5

5층 메모리로 에이전트 학습 관리

Manage agent learning with 5-layer memory

시나리오

Scenario

에이전트가 같은 실수를 반복한다. 패턴으로 끌어올리고, 야간 reflection으로 traits에 반영시켜 다음 세션부터 자동 회피.

An agent keeps making the same mistake. Promote it to a pattern; nightly reflection bakes it into traits so future sessions avoid it.

명령

Commands

$ oxg memory add --kind rule \
    "git push 전 항상 pre-commit 통과 확인" --pin

$ oxg pattern detect --since 7d
[RECURRING] "skipped pre-commit" — seen 4 times

$ oxg reflect --tonight
[L4 trait] cautious-deployer (confidence 0.81)

$ oxg recall --query "deploy 절차"
[rule pinned] git push 전 항상 pre-commit 통과 확인
[trait]       cautious-deployer

기대 결과

Expected result

  • 다음 세션에서 같은 종류 실수가 반복되면 prompt 단계에서 차단됨
  • Next session catches the same mistake at prompt time, before it happens

주의점

Caveats

  • 핀(pin)은 망각 회피 수단이지 맹신은 금물 — 정기적으로 가지치기.
  • Pinning prevents forgetting, but isn't a guarantee — prune regularly.
