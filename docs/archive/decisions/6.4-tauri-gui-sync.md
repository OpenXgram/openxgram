# ADR 6.4 — Tauri GUI 동기 결정

날짜: 2026-05-10 KST

## 상황

Tauri 데스크톱 GUI (`ui/tauri/`) 가 Phase 1 초기에 만들어졌고, daemon 의 `/v1/gui/*` HTTP API 로
원격 호출. 그러나 Phase 1 의 핵심 흐름 (master ↔ 메인 ↔ 서브에이전트) 은 채널 (Discord/Telegram)
관전·참여로 충분히 동작 중. master 는 사실상 GUI 안 씀.

## 6.4.1 — agent runtime 통합 또는 분리 결정

**결정: 분리 유지.**

- agent runtime 은 `xgram agent` (Rust 바이너리) 으로 daemon 과 같은 머신에서 백그라운드 가동.
- Tauri GUI 는 별도 데스크탑 앱 — daemon 의 `/v1/gui/*` 만 통해 read-only 조회 + 일부 mutation.
- agent 와 GUI 가 한 프로세스에 묶이면: webkit2gtk 의존 / 데스크탑 환경 강제 / headless 서버에서 agent 못 띄움.

**Why:** master 의 "한 노드 = 한 에이전트" 원칙은 daemon + agent 가 헤드리스 서버에서 자율 동작해야 함.
GUI 는 사람이 보는 창이지 에이전트의 일부가 아님. "메모리·인프라 vs 자율 에이전트 경제" 정정과 일치.

**How to apply:** agent 기능 (LLM 응답·peer routing·self trigger) 추가 시 무조건 `crates/openxgram-cli/src/agent.rs`
또는 sibling Rust 모듈에. Tauri 측은 `/v1/gui/*` 스키마만 따라 쫓아옴.

## 6.4.2 — 핸들 도입 후 재정렬

**결정: 다음 phase (Phase 3) 핸들 등록 + ENS resolver 안정화 직후, 6.4 의 친구 목록 UI 와 통합 conversation
스레드 뷰를 한 번 정렬한다.**

- 마스터 메모리 (`project_mission.md`): "통합 메신저 허브 — 친구 목록 = OpenXgram peer + Discord 채널 +
  Telegram chat 모두 한 목록에. 클릭 → 그 대상과의 메시지 스레드. 보내기 → 적절한 transport 자동 라우팅."
- 즉 핸들 (Phase 3.1~3.3) 이 들어오기 전엔 통합 친구 목록의 1차 source of truth (handle resolver) 가 없음.
- Tauri 의 친구 목록·대화 뷰는 핸들 안정화 후 redesign 한 번에.
- 그때까지 Tauri 는 현재 read-only + peer/schedule mutation 수준 유지 (release 동봉).

**Acceptance (Phase 3 완료 시점에 재검토):**
1. `xgram find @<handle>` 가 안정적 결과 반환
2. Tauri 친구 목록이 (peer + Discord channel + Telegram chat + handle resolved) 한 view 로 표시
3. 메시지 send 가 발신자 종류별 자동 routing (peer_send / Discord webhook / Telegram sendMessage)

본 결정은 **release-ceremony 함정 회피** — UI 다듬기보다 메시징 본질 (cross-node + 핸들) 먼저.
